# Governance

This document explains how repository stewardship, contributions, and future
license decisions are handled for X-Hub System.

It is a project-governance document. It is not a separate software license and
does not override `LICENSE`, `LICENSE_POLICY.md`, or `TRADEMARKS.md`.

## Current Model

- The repository is currently source-available under `FSL-1.1-MIT`.
- Each version converts to MIT on the second anniversary of the date that
  version was made available, as stated in `LICENSE`.
- The project owner retains control over release scope, roadmap, branding,
  trademark use, and future licensing decisions for later versions.

## Contributions

External contributions are welcome.

Contributions are reviewed based on:

- technical quality
- security and trust-boundary impact
- scope alignment with the validated public mainline
- maintenance cost
- documentation and test coverage when behavior changes

Acceptance of a contribution does not create maintainer, ownership, or
governance rights for the contributor beyond the rights granted by the
repository license and applicable law.

## Future License Direction

The project may evaluate broader or more permissive licensing for future
versions if the repository reaches sustained community health and the business
risks are acceptable.

Signals that may inform that decision include:

- sustained high-quality external contribution
- healthy review and security-reporting behavior
- maintainable release and support load
- clear protection against direct free-riding on the core product
- stable project governance and release discipline

There is no automatic contributor-count trigger for relicensing.

In particular, the project does not promise that reaching any fixed number of
contributors will cause the repository to switch immediately to MIT.

Any future license change for later versions, if it happens, will be announced
explicitly in repository documents and release materials.

## Stewardship

The project owner may:

- accept or reject contributions
- define release boundaries
- choose what is public by default versus intentionally held back
- revise governance processes
- revise licensing for future versions

Those decisions should be communicated clearly in top-level repository
documentation when they materially affect contributors or adopters.
