# Hub Receipt — Unified Envelope Specification v0.1 (Draft)

**Status:** Draft, pre-coordination with mcp-trust-registry v0.1 and agent-2fa v0.1
**Editor:** X-Hub-System project (reference implementation)
**Date:** 2026-06-25
**License of this document:** CC BY 4.0

---

## Abstract

Hub Receipt defines a minimal, signed JSON envelope for execution evidence that any X-Hub-aligned tool can emit and any party can verify. It is intentionally **not** a payload format, **not** a transport, and **not** JSON-LD in v0.1. Consumer specifications (notably mcp-trust-registry session receipts and agent-2fa authorization receipts) extend the envelope by populating a domain-specific `claims` object; the envelope itself defines only the fields needed to sign, identify, and verify a receipt.

The protocol is intentionally minimal; complexity is deferred to future versions only when forced by interoperability needs.

## 2. Terminology

The key words MUST, MUST NOT, SHOULD, SHOULD NOT, and MAY in this document are to be interpreted as described in RFC 2119.

- **Issuer.** The party that signs a receipt. Identified by `issuer_key_id`.
- **Subject.** The thing the receipt is about — an MCP server, an agent action, or any other domain-specific entity.
- **Claim.** A single key/value entry inside the `claims` object. Semantics are defined by the consumer specification, not by this document.
- **Envelope.** The outer JSON structure defined in §3, common to all Hub Receipts.
- **Signed Receipt.** An envelope whose `signature` has been computed over the canonical bytes of the envelope minus the `signature` field, per §4 and §5.
- **Consumer Spec.** A specification (e.g. mcp-trust-registry, agent-2fa) that defines the `subject` domain and `claims` schema for one class of receipt.

---

## 3. Envelope format

An envelope is a JSON object with the following required fields:

```json
{
  "schema": "hub-receipt/envelope/v0.1",
  "receipt_id": "rcpt-abc123def456",
  "issued_at": "2026-06-25T12:00:00Z",
  "issuer_key_id": "ed25519:abcdefghijklmnopqrstuvwxyz234567abcdefghijklmnopqrst",
  "subject": { "domain": "hub-receipt-test", "name": "example" },
  "claims": { "hello": "world" },
  "signature": "ed25519:<base64 payload>"
}
```

| Field | Type | Constraint |
|---|---|---|
| `schema` | string | MUST equal `hub-receipt/envelope/v0.1` for this version. |
| `receipt_id` | string | MUST match `^rcpt-[a-f0-9]{6,32}$`. Receipt-unique within an issuer. |
| `issued_at` | string | RFC 3339 date-time, UTC (`Z` suffix RECOMMENDED). |
| `issuer_key_id` | string | MUST match `^ed25519:[a-z2-7]{52}$`. Identifies the signing key per §5. |
| `subject` | object | MUST have a `domain` (string). MUST have exactly one of `name` (string) or `identifier` (string). |
| `claims` | object | MUST be a non-empty JSON object. The envelope does NOT constrain inner shape; consumer specs do. |
| `signature` | string | MUST match `^ed25519:[A-Za-z0-9+/]+={0,2}$`. Computed per §5. |

The envelope MUST NOT carry any additional top-level fields in v0.1. Consumer-specific data MUST live inside `claims`.

---

## 4. Canonicalization

For hashing and signing, the canonical form of an envelope is its JSON representation with the same rules as mcp-trust-registry §4.4: UTF-8 encoding without BOM, lexicographically sorted keys at every depth, no whitespace between tokens, numbers in their shortest unambiguous form, Unicode strings in NFC.

To produce the canonical bytes for signing, the `signature` field MUST be removed from the object before canonicalization. After signing, the `signature` field is reinserted at the top level; readers MUST tolerate it being placed anywhere in serialized form because they will re-canonicalize before verification.

---

## 5. Signing

Signatures are ed25519. Encoding:

- `issuer_key_id` is the prefix `ed25519:` followed by the 32-byte ed25519 public key encoded as 52 lowercase base32 characters without padding.
- `signature` is the prefix `ed25519:` followed by the 64-byte ed25519 signature encoded as standard base64 with `=` padding (88 characters total payload).

Signing input is the canonical bytes from §4. v0.1 signs canonical bytes directly; no pre-hash. SHA-256 MAY appear inside `claims` for consumer-spec-defined hashes (e.g. `manifest_hash`), but is not part of the envelope's signing algorithm.

Keyless / Sigstore signing is out of scope for v0.1; it is identified as a v0.2 candidate in §9.

## 6. Verification

A verifier MUST perform the following steps in order:

```
1. parse JSON; reject if not a JSON object.
2. validate envelope shape against the schema in §3 (all required fields,
   correct patterns); reject on any failure.
3. read issuer_key_id; resolve to an ed25519 public key via a
   consumer-spec-defined directory (NOT specified here).
4. remove the `signature` field; produce canonical bytes per §4.
5. ed25519_verify(public_key, canonical_bytes, signature); reject on failure.
6. read subject.domain; pass `claims` to the consumer spec validator for
   that domain. The consumer spec MUST own all claims validation.
```

A verifier that succeeds through step 5 has established that the envelope was signed by the holder of the key identified by `issuer_key_id`. It has **not** established that the claims are truthful — only that the issuer asserted them.

## 7. Extension by consumer specs

This document defines only the envelope. Consumer specifications fill in the domain-specific shape of `subject` and `claims`.

**mcp-trust-registry session receipt.** A session receipt extends the envelope with `subject = { "domain": "mcp-trust", "name": <server_name> }` and `claims` carrying `session_id`, `manifest_hash`, `artifact_hash`, `started_at`, `ended_at`, `tool_calls`, `capability_denials`, `policy_hash`, and optionally `denial_summary`. The `issuer_key_id` is the verifier's signing key. mcp-trust-registry v0.1 currently inlines these fields at the top level of its own receipt schema rather than nesting them under `claims`; the two forms are structurally compatible and a future harmonization pass MAY collapse them.

**agent-2fa authorization receipt (forward reference).** An authorization receipt extends the envelope with `subject = { "domain": "agent-2fa", "identifier": <action_id> }` and `claims` carrying `risk_level`, `modality` (e.g. `touch-id`, `voice`, `push-confirm`), `authorizer_id`, `granted_at`, `expires_at`, and `policy_ref`. The `issuer_key_id` is the authorizer device's signing key. Replay protection lives in the consumer spec via `expires_at`; the envelope itself is replay-agnostic.

Additional consumer specs MAY register new `subject.domain` values and define their own `claims` schemas without modifying this envelope.

---

## 8. Security considerations

Hub Receipt binds a `claims` set to an issuer's signing key. It does **not** certify that the claims are true; that is the consumer specification's domain logic. A consumer spec MUST define how to interpret claim falsity (e.g. mcp-trust treats a mismatched `manifest_hash` as a verifier bug; agent-2fa treats an expired `expires_at` as a non-authorization).

Replay defenses live in the consumer specification. The envelope's `issued_at` is informational; a consumer spec that needs replay protection MUST embed `expires_at`, nonce, or session-binding fields in `claims`.

Key revocation is out of scope at the envelope layer. Consumer specs MAY define their own revocation lists keyed on `issuer_key_id`.

## 9. Open questions

- JSON-LD `@context` for v0.2 to enable machine-discoverable schema URIs and linked-data interoperability.
- Sigstore Rekor (or equivalent transparency log) integration as an optional `transparency_log_entry` field at the envelope layer.
- Key rotation and revocation strategy: per-spec lists vs an envelope-layer revocation directory.
- Multi-issuer countersign (envelope with N signatures) for high-assurance receipts.
- Receipt aggregation via Merkle batching, primarily for cost reduction when emitting many small receipts.
