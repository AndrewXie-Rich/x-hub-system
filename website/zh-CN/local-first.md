# 本地优先

<p class="lead">
敏感代码留本地。重推理打到 Claude 或 GPT。两者共享同一个 Hub 治理的记忆和审计——不是因为它们是同一个产品,而是因为它们在同一个 Hub 后面。本地优先在这里不只是"跑本地模型",而是控制平面留在你手里。
</p>

<div class="preview-note">
  <strong>本地不是"全部"或"全无"。</strong>
  真实场景大多是混合的:个人 / 敏感上下文走本地模型,任务能力需求大时切付费模型。X-Hub 让混合变得连贯——一个额度视图、一套 fallback 策略、一条审计链。
</div>

## 一套受控平面，同时覆盖本地与付费模型

同一套 Hub 可以统一治理：

- 本地模型
- 付费模型供应方
- 本地多模态运行时
- 项目级路由与能力姿态

这点很重要，因为很多系统会把本地路径和付费路径做成两个完全不同的产品世界。

## 纯本地模式能带来什么

如果 Hub 跑在用户自有硬件上，并尽量只走本地模型：

- 核心推理路径可以脱离第三方云基础设施
- 控制平面不需要寄居在 SaaS 默认形态里
- 外部模型凭据可以从核心路径中消失
- prompt 和上下文外发可以明显减少

这不等于“所有威胁都没有了”，但它确实是隐私、控制权和运行独立性上更强的起点。

## 本地运行时产品化

本地方向已经不只是粗糙试验桥接，而是在朝更完整的产品面推进：

- 能力包清单和真实状态表达
- 运行前的兼容性判断
- 更清楚的导入指导和恢复反馈
- quick check 与 bench 类表面
- 让本地 embedding、语音、视觉和 OCR 都处在同一 Hub 姿态下

## 为什么“能力包真相”重要

多模态本地 AI 系统常见的问题之一，是看起来“像可用”，实际上依赖、模型包或运行环境并没有准备好。

因此系统需要更诚实地表达：

- 什么已经安装
- 什么被禁用
- 什么不兼容
- 什么现在可以测试
- 什么应该 fail-closed

## 推荐部署形态

对很多个人、团队和工作室部署来说,更清晰的形态是把 Hub 当作一台长期在线的可信控制平面,而不是临时开在某个项目终端里的进程。

推荐硬件可以从这类形态开始:

- Mac mini:适合作为常开 Hub、家庭 / 小团队控制平面、本地轻量模型和远程配对入口
- Mac Studio:适合更重的本地模型、多模态运行时、更多项目并行和更长时间的 Supervisor 运行
- 用户自己的 MacBook:适合开发、演示和单人试用,但不一定是长期运行 Hub 的最佳位置
- Linux box via `docker-compose up`(收口中,90-day P0——见[状态与路线图](/zh-CN/status-roadmap))

推荐运行方式:

- Hub 跑在用户自有 Apple silicon 硬件或 Linux box 上
- 真正 ready 的本地运行时再开启
- 付费模型按需、选择性接入,而不是默认放开
- X-Terminal、远程入口和 skill 都通过 Hub 授权,不直接成为最终控制面
- 运行时真相、模型状态、额度、记忆和审计始终对用户可见

## 当你的 IDE agent 用本地模型时,X-Hub 仍然夹在前面

这一条很重要,因为"本地模型"问题不只是 X-Hub 自己的 runtime。到 2026 年,主流 IDE agent 大多接受本地后端:

- Cursor 可以接本地 Ollama endpoint
- Claude Code 配好后可以包本地 MLX 模型
- Cline / Aider / Continue 都支持本地 provider

它们这样接的时候,**X-Hub 仍然夹在 agent 和它要做的动作之间**。模型在本地跑不改变信任边界——Hub 仍然是记忆写入、技能安装、破坏性动作、审计汇聚的地方。

这样才能做到本地优先,而不是嘴上本地、控制权却仍然漂在外面。
