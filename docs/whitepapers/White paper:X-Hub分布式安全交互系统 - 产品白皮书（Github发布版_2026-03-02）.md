# 摘要（Executive Summary）

X-Hub系统的核心结论只有一句话：让“不可信终端”也能安全执行“可信现实动作”。

传统AI产品强调“会回答”，X-Hub强调“可托付执行”：即使终端被攻破，关键动作仍需经过Hub签名、跨终端校验和授权门禁，系统仍可保持可控、可追溯、可冻结。

## 一分钟抓重点

- 唯一可信核心（X-Hub）：AI推理、联网代理、授权审批、审计、记忆治理统一收敛到Hub，终端默认不参与安全决策；
- 多AI能力统一治理：本地模型与付费模型都走Hub控制面，能力调用受grants/quota/audit/kill-switch约束；
- 自动化不打断：默认自动执行；高风险能力通过Pre-Grant（默认TTL=2h）提前收集授权，减少执行时打断；
- 跨终端可验证：高风险动作使用Hub签名Manifest + 一次性校验码（SAS），确认端独立验签、独立算码，防止单端被劫持造成误操作；
- 多项目并行可管控：X-Terminal Supervisor可并行管理多个Project，按heartbeat汇报“进度/风险/待授权/下一步建议”；
- 记忆与价值双护栏：五层内存架构提升长期执行稳定性，X-Constitution（价值宪章）提供不可绕过的行为约束；
- 效率工程化：多泳池拆分 + 反阻塞续推 + 三段式提示词（Stable Core + Task Delta + Context Refs），在质量可控前提下降低等待与Token浪费。

# 一、产品概述

## 1.1 产品定位

X-Hub系统是以X-Hub为唯一可信核心、终端为轻量化交互载体，融合集中式AI能力与多网络模式适配性的分布式安全交互系统，核心定位为“金融级安全AI交互可信管控平台”。

系统打破“安全与体验互斥”的困境，通过集中式可信管控与分布式轻量化交互相结合，实现AI对话流畅度、多场景适配性、资产安全性的三重平衡，适配家庭、中小企业、高安全需求场景（如离线AI）等各类使用场景。

## 1.2 核心价值

- 安全可控：X-Hub为唯一可信节点，终端默认不可信，全链路加密+沙箱隔离，坚守金融级安全底线；

- 可信执行：关键动作采用Hub签名Manifest + SAS + 授权门禁，做到可验证、可追溯、可冻结；

- 多AI统一治理：本地模型与付费模型统一纳入Hub能力面，支持授权、配额、审计、撤销与成本控制；

- 长期稳定记忆：五层记忆（Raw/Observations/Longterm/Canonical/Working Set）分层治理，降低上下文漂移；

- 价值行为约束：X-Constitution以固定约束（pinned）+触发式注入方式保障“能做不等于该做”；

- 效率与成本协同：多泳池并行、反阻塞自动续推、三段式提示词共同提升吞吐并降低Token浪费；

- 多场景适配：支持离线、局域网、远程公网访问等4种运行模式，灵活满足不同网络环境需求；

- 灵活扩展：终端类型可按需适配，Skills/Connectors/开源兼容能力可迭代升级。

## 1.3 核心设计原则

1. X-Hub集中可信原则：所有AI推理、授权审批、审计、记忆治理统一收敛到Hub，终端默认不参与安全决策；

2. 终端不可信原则：终端仅承担输入输出与轻量缓存，不持有核心密钥、不保有主数据，被入侵后不应破坏Hub安全边界；

3. fail-closed优先原则：证据不足、Gate未过、依赖未绿、授权不完整时，系统默认阻断，不以“先跑起来”替代安全；

4. 证据优先（Evidence-First）原则：关键结论必须可机读、可追溯、可回放，Gate/KPI/审计引用优先于口头结论；

5. 确定性编排原则：同输入、同档位、同约束应得到可复现的拆分与调度结果，避免协作漂移；

6. 最小权限与隔离原则：按`device/project/thread`与泳道边界隔离权限，禁止跨泳道直接改写状态；

7. Token效率原则：提示词采用“三段式（Stable Core + Task Delta + Context Refs）”，以最小充分上下文驱动高质量执行；

8. 可回滚演进原则：所有关键链路必须定义回滚点与降级策略，确保系统在高并发与高风险场景下仍可控、可恢复。

# 二、核心架构：X-Hub五层内存架构（✅基础能力已落地，🟡高级层持续迭代）

X-Hub内存系统采用“五层递进式”架构，实现内存从“证据层”到“注入层”的结构化治理，既保障全流程可追溯，又平衡AI交互的Token效率，同时承载X章程（X-Constitution）价值宪章的安全约束，构建“技术安全+价值安全”双重防护。

## 2.1 五层架构全面解析

|层级|定位|核心特性|数据来源/处理逻辑|
|---|---|---|---|
|1. 原始存储层（Raw Vault）|证据层（无限存储）|仅追加存储、不可篡改、默认不注入，作为审计/追溯的唯一事实来源|完整交互轮次、工具输出、文件摘要/哈希；按时间段归档压缩，保留完整证据链|
|2. 观测层（Observations）|可搜索结构化层|从原始存储层提取结构化内存（事实/偏好/约束/决策/经验），支持全文检索/向量/时间线搜索|由X-Hub专属AI角色自动提取，按`device/project/thread`隔离，支持精准检索|
|3. 长期内存层（Longterm Memory）|文档型长期内存层|聚合观测层内容形成结构化文档（目标/架构/约束/价值宪章），渐进式披露|由AI角色维护升降级；默认注入摘要，用户查询时补充细节+证据链接；价值宪章永不下沉|
|4. 规范内存层（Canonical Memory）|精简注入层|固化核心键值对（偏好/短期目标/接口协议），按范围隔离，默认注入|从观测层/长期内存层升级而来，数量少、稳定性高，Token效率最高|
|5. 工作集层（Working Set）|短期上下文层|缓存最新N轮对话，默认注入，超出预算时丢弃最旧内容|终端/X-Hub双缓存；终端侧缓存仅用于崩溃恢复，核心内容同步至X-Hub，按`device/project/thread`隔离|
工程状态补充（2026-03-02）：
- ✅ Raw Vault / Canonical / Working Set主链路可用；
- 🟡 Observations / Longterm正在从规范到实现持续补齐（含FTS/向量/时间线与渐进披露细化）；
- 🟡 记忆健康检查与降级策略已定义，正向统一报告与自动修复闭环推进。

## 2.1.1 X章程（X-Constitution）价值宪章（全局AI安全与价值约束）

作为全局AI安全与价值约束，X章程（价值宪章）置于X-Hub五层内存架构的第3层（长期内存层），标记为固定常驻（pinned）、永不下沉、不可篡改的顶层L0规则条目，与长期内存层作为文档型长期约束层的定位天然适配。

该宪章由X-Hub统一制定、存储、管理，仅可通过X-Hub冷存储Token授权更新，任何终端（含X-Terminal）均无修改、删除、绕过权限。同时，该宪章仅对使用X-Hub内存系统的X-Terminal生效：X-Terminal在处理项目（Projects）、进行多轮对话、调用技能（Skills）、执行AI决策时，必须从X-Hub的长期内存层读取并严格遵循X章程，将其作为全局行为约束与不可突破的安全底线；普通通用终端不使用X-Hub内存、不同步上下文至X-Hub，因此不会加载及受该价值宪章约束。

为保障系统高效运行，X章程采用极致轻量化、触发式注入设计：仅保留最核心的价值规则与安全约束，内容固定、体积小巧；采用按需触发注入而非全量常驻上下文，不会增加系统负担、产生大量额外Token消耗，对AI推理速度、终端运行性能、上下文长度无明显影响。

X章程在底层架构层面不可或缺：一是防止人类通过指令构造、上下文注入等方式诱导AI执行恶意、违规、未授权操作；二是为未来AGI自主决策、多项目协作、技能自动升降级与执行提供不可动摇的价值边界；三是从“价值安全”层面补充技术防护短板，与AI沙箱、权限控制、内存隔离共同构成完整的安全防御体系，从根本上防范恶意利用与行为失控风险。

更准确地说，X章程的目标，是把正确价值观刻进AGI的“行为基因”里。这里的“基因”并非修辞性口号，而是指一组高于具体任务目标的系统级约束：固定常驻、受授权更新、可触发注入、可策略校验、可审计追踪，并能在高风险、无授权或不确定场景下直接阻断执行。它不是上线之后不断追加的安全补丁，而是要求系统从一开始就带着价值边界运行。

因此，X章程并不承诺“消灭一切风险”，而是把现实Agent已经暴露出的典型失控路径前置为架构红线，用于避免出现以下情况：

- 提示词注入风险：攻击者在网页、邮件、文档或其他外部内容中埋入隐藏恶意指令，诱导Agent在读取后泄露本地密钥、系统凭证或敏感数据；
- 误操作风险：Agent错误理解用户意图，在缺乏明确范围、对象与影响确认的情况下，直接删除电子邮件、核心生产数据、重要文件或执行其他不可逆操作；
- Skills投毒风险：恶意或高风险第三方插件在导入后窃取API Key、读取隐私文件、植入木马后门，甚至把设备变成外部控制节点；
- 安全漏洞放大风险：当底层实现存在高中危漏洞时，攻击者进一步利用漏洞完成越权访问、隐私窃取、代码仓库与业务数据泄露，甚至拖垮整个业务系统。

X章程在这里的作用，不是单独“魔法式解决一切”，而是要求这些风险在系统层面先被视为高危与默认不可信：外部输入不能直接改写系统目标，破坏性动作不能跳过确认，第三方Skills不能天然继承高权限，异常与漏洞场景下必须尽量收缩权限、保留审计并优先fail-closed。换句话说，它的目标就是尽可能避免这些情况发生，即使无法百分之百消灭风险，也要让系统更难失控、更难被诱导、更难被静默攻破。

## 2.2 内存升降级机制

内存的流转由X-Hub内置专属AI角色全生命周期管理，无需人工干预（除非特殊配置），核心规则如下：

- 升级路径：原始存储层 → 观测层（自动提取）→ 长期内存层（聚合文档）→ 规范内存层（固化核心事实）；

- 下沉规则：
  

    - 观测层/长期内存层：自动合并、汇总、标记过期，保留证据指针（turn_id/时间范围/摘要哈希）；

    - 原始存储层：仅做存储治理（按时间归档压缩），不影响检索，索引永久可追溯；

- 特殊规则：X章程（价值宪章）作为固定常驻L0内存，永不下沉，仅在触发词/高危场景下按需（中英文双语）注入。

## 2.3 内存核心技能（Memory-Core Skill）：内存系统的“核心规则引擎”

内存核心技能（Memory-Core Skill）是X-Hub内存处理的唯一指导规则集。产品层可继续沿用 `Skill` 命名，但在实现边界上，它应被理解为 **X-Hub 内建的受治理规则资产**，而不是一个拥有全写权限的单体执行 AI。memory 使用哪个 AI 由用户在 X-Hub 中显式选择；X-Hub 负责调度与审计；最终 durable 落库仍由 Writer/Gate 决定。核心设计如下：

- 不可篡改：运行期间不可动态修改，终端及普通技能（Skills）均无修改权限；

- 唯一修改入口：仅可通过X-Hub冷存储中存放的X-Hub Token认证后修改，所有修改记录全审计；

- 核心职责：
  

    - 指导五层内存架构的流转规则（提取/聚合/固化/注入）；

    - 控制内存的隔离维度（device/project/thread）；

    - 定义内存脱敏规则（如含<private>标签内容默认丢弃/脱敏）；

    - 约束 Memory Scheduler / Worker / Writer 的角色边界，以及相关技能（Skills）的调用时机与优先级，保障“无缝调用”；

    - 控制X章程的注入时机、权限及生效范围。


## 2.4 为什么“记忆 + X章程”是产品护城河

- 长期任务不会“越做越乱”：五层内存把证据、结构化事实、长期文档和注入层分开治理，避免上下文污染；
- 多终端协作不会“越权漂移”：终端可随时换，约束与事实都在Hub，策略不依赖某个设备本地状态；
- 自动化增强不会“价值失控”：X章程作为长期约束在关键场景触发，确保“能做”不等于“该做”；
- 这不是只提升回答质量，而是提升“长期可托付执行能力”：对家庭、企业、机器人场景都成立。

# 三、技能（Skill）升降级与分类体系

X-Hub内存系统的核心扩展是“内存→技能（Skills）转化”，技能作为内存的可复用单元，按“分类、隔离、自动+人工确认”原则管理，同时受X章程价值宪章的全局约束。

## 3.1 技能（Skill）核心隔离规则

技能的隔离粒度为“一台设备×一台X-Terminal×一个项目=独立技能域”，不同域的技能完全隔离，不可跨域调用、不可相互污染，与内存系统的三维隔离原则保持一致。

## 3.2 技能（Skill）升级规则

技能由内存/交互事件触发成为候选技能，按置信度分级升级，所有升级过程均需遵循X章程约束：

1. 自动升级：候选技能置信度超过预设阈值（可配置），由X-Hub自动升级为可用技能；

2. 人工确认：置信度不足的候选技能，推送至终端用户确认，经用户确认后方可升级；

3. 审核编辑：所有技能的审核、编辑操作在X-Terminal端完成，最终存储于X-Hub端，编辑记录全审计，编辑内容需符合X章程规则。

## 3.3 X-Hub端技能（Skill）分类体系

X-Hub端按复用范围将技能分为三类，确保按需调用、无冗余、无未授权访问，均受X章程约束：

|技能类型|定位|管理方式|调用规则|
|---|---|---|---|
|内存核心技能（Memory-Core Skill）|系统级核心规则资产|X-Hub专属，仅可通过冷存储Token修改，全X-Hub通用；不按普通技能权限模型执行|最高优先级的受治理规则资产；约束 Scheduler/Worker/Writer，不等于单体执行AI|
|通用技能（General Skill）|跨项目复用|可通过X-Terminal“导入按钮”从外部导入，由X-Hub统一管理|项目授权后可调用|
|单项目专属技能（Single-Project Exclusive Skill）|仅对应项目可用|由该项目的交互内存生成，在X-Terminal端编辑，存储于X-Hub|仅在所属项目内自动调用|
## 3.4 技能“无缝调用”保障机制

为确保技能在需要时精准调用，X-Hub内置调度逻辑，同时遵循X章程约束：

1. 调用触发条件：根据当前交互上下文、项目标识、设备/终端权限自动匹配；

2. 治理优先级规则：内存核心技能（规则资产）约束 Scheduler/Worker/Writer，且不进入普通 skill runner 的冲突解析链；普通可执行技能的解析顺序仍为“通用技能 > 单项目专属技能”；

3. 无冲突保障：按`device/project/thread`隔离域匹配技能，跨域技能不可见；

4. 实时同步：X-Terminal端编辑的技能实时同步至X-Hub，调用时优先使用最新版本，同步过程受X-Hub安全审计。

## 3.5 多AI能力管理（本地/付费）与能力门禁

在X-Hub体系中，“模型选择”不是终端本地行为，而是受策略约束的系统能力：

1. 统一路由：Worker/Supervisor可按角色映射本地模型或付费模型，统一走Hub能力面，避免终端私自持有密钥；

2. 授权策略：付费模型能力支持“首次人工一次性授权，后续在策略与配额内自动续签/自动放行”；

3. 能力门禁：模型调用、联网工具、外部动作统一纳入grant主链，缺授权时返回machine-readable deny_code并fail-closed；

4. 成本与风控联动：通过token cap/cost cap/ttl/daily cap约束高成本能力，异常可由Kill-Switch即时冻结；

5. 回退策略：在策略允许且安全边界满足时，可从远程能力降级到本地能力以保持可用性；高风险链路仍以授权与证据门禁为先。

# 四、运行模式

X-Hub系统支持4种运行模式，覆盖离线、局域网、远程公网访问等全部场景，按安全等级从高到低排序（风险从低到高相反），各模式独立适配、可同时运行，均由X-Hub统一管控，且均遵循核心安全原则及X章程约束（仅对X-Terminal生效）。

## 4.1 模式0：纯离线局域网（安全等级：最高）

### 4.1.1 连接特性

终端与X-Hub处于同一私有局域网，X-Hub无公网访问能力，终端也无互联网访问权限，终端与X-Hub的通信仅可通过局域网实现，属于物理隔离级场景。

### 4.1.2 安全风险

攻击面极小，仅存在两类潜在风险：设备被物理接触；内网中被入侵设备的横向移动。无远程攻击可能，上下文、内存、通信均在内网闭环，安全可控。

### 4.1.3 适用场景

最高保密级场景、无需联网及远程访问的离线AI交互场景，如机密AI推理、离线语音交互等。

## 4.2 模式1：安全局域网（安全等级：高，推荐日常使用）

### 4.2.1 连接特性

终端与X-Hub处于同一私有局域网，终端无公网访问能力，X-Hub可通过桥接器（Bridge）实现可控公网连接（仅用于AI联网需求，如付费模型调用、联网推理），终端仅可通过局域网与X-Hub通信。

### 4.2.2 安全风险

风险较低，终端被入侵后，无法连接外部控制端、下载恶意程序、扩散攻击，即便获取X-Hub内网IP，也无法将攻击渗透至外网，不影响X-Hub核心安全。

### 4.2.3 适用场景

日常AI交互场景，如家庭AI对话、企业内部AI办公，需AI具备联网能力但希望终端安全可控，避免终端遭受公网攻击。

## 4.3 模式2：普通局域网（安全等级：中）

### 4.3.1 连接特性

终端与X-Hub处于同一私有局域网，X-Hub具备公网访问能力，终端可独立访问公网（用于非AI日常场景，如网页浏览、办公软件使用），终端也可通过局域网与X-Hub通信。

### 4.3.2 安全风险

风险中等，终端为主要攻击入口——终端可通过网页、邮件、漏洞、木马被入侵，入侵后可扫描内网获取X-Hub内网IP，尝试对X-Hub发起内网攻击、重放攻击、伪造请求，风险根源在于终端不可控。

### 4.3.3 适用场景

对终端管控要求较低的局域网场景，如中小企业办公、家庭混合使用（部分终端需公网访问），需结合防火墙、虚拟局域网（VLAN）缩小攻击面。

## 4.4 模式3：远程公网加密连接（安全等级：中高）

### 4.4.1 连接特性

终端与X-Hub无物理局域网关联，通过WireGuard/ZeroTier/Cloudflare Tunnel等加密专用通道实现远程公网互联，X-Hub保留独立公网访问能力，终端可远程访问X-Hub的AI能力。

### 4.4.2 安全风险

该模式是现实中最常用的远程工作形态之一。与局域网模式相比，主要风险来自“公网暴露面”而非“终端可信性”：

- 公网可见性：隧道/中继/域名在公网可被扫描与探测，可能遭受DDoS与暴力探测；

- 凭据风险：证书/Token泄露会让攻击者以该设备身份访问X-Hub服务（影响范围取决于该设备的授权范围与可见数据）；

- 协议与配置风险：隧道协议/中继服务自身漏洞或错误配置可能扩大攻击面。

但X-Hub的安全模型从设计上假设“终端会被攻破”。因此在终端被黑/设备被劫持时：

- Hub核心不会因终端状态而被接管；高风险动作仍受Hub侧授权、配额、审计与Hub签名Manifest约束；

- 若该终端使用的是Hub提供的AI能力，Hub可立即冻结该设备：停止其AI调用、停止联网/付费能力、撤销Grant/Pre-Grant，并触发设备级或全局Kill-Switch；

- 需要注意：凭据被盗可能导致该设备可见的数据被读取/外泄，因此必须配合短生命周期凭据、轮换、最小权限与快速吊销机制。

### 4.4.3 适用场景

常态远程办公/出差/多地点协作场景。推荐组合启用：强身份（客户端证书+Token）、设备级权限隔离、访问源限制（allowed_cidrs/可选地理围栏）、速率限制与DDoS防护、凭据轮换与一键吊销（Kill-Switch）。

## 4.5 模式风险排序（从低到高）

模式0（纯离线局域网）＞模式1（安全局域网）＞模式2（普通局域网）＞模式3（远程公网）

# 五、终端设计

终端作为系统的用户交互入口，默认不可信，按功能定位分为两类，均采用“轻量化设计、最小权限”原则，无X-Hub核心安全权限，仅负责输入、输出及基础交互，具体设计如下：

## 5.1 普通通用终端

### 5.1.1 定位

轻量化、通用型交互终端，适用于无需连续交互体验的基础AI对话场景，追求极致轻量化安全。

### 5.1.2 功能边界

- 核心功能：仅接收用户输入（如AI对话提问），自行保存对话上下文，调用X-Hub的AI推理能力获取推理结果并同步展示给用户；不转发任何对话内容至X-Hub，X-Hub仅提供AI推理支持，不处理普通终端的对话；

- 权限限制：无复杂本地交互逻辑；不执行任何AI模型推理（仅调用X-Hub的AI推理能力，付费AI的API密钥统一存储在X-Hub端，终端无获取权限）；终端可自主决定是否开启文件读写、系统执行、工具执行等高危操作及设备自身的系统级指令，此类操作仅作用于终端自身，与X-Hub无关联，X-Hub不干涉终端自身系统操作；此外，普通终端与X-Terminal的Token使用方式、用量均由X-Hub统一设定，X-Hub可全面控制模型的联网能力，终端无独立控制权限。

- 安全特性：仅保存自身对话的上下文，无其他数据残留；不转发任何对话至X-Hub，X-Hub也不处理其对话，因此被入侵后仅可篡改本地保存的上下文及展示内容，无法影响X-Hub，也无法泄露其他终端的历史信息，风险可控。

## 5.2 X-Terminal

### 5.2.1 定位

具备连续AI交互能力的增强型终端，适用于需要长期对话、连续体验、多项目管理的场景（如连续AI咨询、多轮交互、多项目并行管控），依赖X-Hub实现内存管理、安全控制、项目数据同步，详细设计参见《X-Terminal白皮书》。

### 5.2.2 功能边界

- 本地缓存：仅缓存最新3-5轮对话上下文，用于界面流畅与崩溃恢复；不作为主数据，不参与安全决策；

- 上下文同步：每轮新输入/新输出都尽力同步至X-Hub；终端不可覆盖、篡改、伪造Hub中的历史与审计；

- 内存依赖：X-Terminal连续交互完全依赖Hub返回的内存视图，终端仅消费不主导，且处理项目时必须遵循X章程；

- Supervisor编排：支持按复杂度自动拆分`pool -> lane`，并按`conservative|balanced|aggressive`档位调节并发粒度；

- 多AI能力接入：同一项目可按任务类型混用本地模型与付费模型；模型调用统一走Hub能力面，受grant/quota/audit约束；

- 用户介入等级：支持`zero_touch`（最少打断）、`critical_touch`（关键打断，默认）、`guided_touch`（高介入）三档协作；

- 提示词与Token优化：泳道执行提示词统一采用三段式（Stable Core + Task Delta + Context Refs），只传增量与引用，降低token浪费；

- 自动推进与解阻：支持heartbeat巡检、阻塞依赖图维护、dual-green依赖判定与unblock路由，blocker转绿后可即时指导等待泳道续推；

- 多项目并行处理：支持多项目并发推进与隔离治理，项目间上下文与权限严格隔离，避免跨项目污染；

- 机器人形态扩展：可作为具身终端执行线下任务（如买水、巡检），证据先回Hub再进入授权与执行闭环；

- 权限硬边界：终端无Hub系统级权限；高风险动作无授权一律阻断；require-real路径禁止synthetic证据冒绿；禁止跨泳道直接改写他人状态。

## 5.3 两类终端通用要求

- 均采用加密连接方式，推荐仅通过域名连接X-Hub；终端侧不固定、不长期存储明文IP（可配置为强制）；

- 均需通过X-Hub身份认证（设备UUID + Token，可选客户端证书）方可接入系统；

- 均无X-Hub操作权限、文件访问权限、网络绕过权限，仅作为轻量化交互载体；

- 均无X-Hub的高危操作权限、系统级操作权限，仅作为轻量化交互载体；两类终端均可执行自身设备的系统级指令，仅作用于终端自身，X-Hub不干涉终端自身系统操作；此外，两类终端的Token使用方式、用量及模型的联网能力，均由X-Hub统一设定、全面控制，终端无独立调整权限。

# 六、安全体系（✅认证/传输/审计已落地，🟡存储加密与防篡改链路持续建设）

X-Hub系统以“可验证、可审计、可回滚”的金融级工程安全为核心目标，构建覆盖身份认证安全、传输安全、存储安全、操作安全、应急安全五大维度的安全体系，并以“Hub签名Manifest + 跨终端校验码（SAS）”作为高风险动作的统一可验证机制，结合X章程价值约束，保障系统与数据安全。

## 6.1 身份认证安全

- 三重认证机制：终端接入X-Hub时，需通过“设备UUID + 专属Token + 可选客户端证书”三重认证，缺一不可，防范非法设备接入；

- 权限分级：按终端类型、使用场景分配不同权限（如普通终端仅可调用AI能力，X-Terminal可同步上下文、读取X-Hub内存），无冗余权限；

- 授权粒度：付费能力按`(device_id,user_id,app_id)`绑定授权；默认“首次人工一次性授权，后续自动续签”，用户可随时撤销；

- 预授权机制：终端调用高危能力（如联网工具、外发动作）时，可由X-Hub提前发放Pre-Grant（默认TTL=2h）与配额限制，过期自动回收；

- 能力门禁主链：高风险能力统一走`ingress -> risk classify -> policy -> grant -> execute -> audit`，缺grant/过期grant/请求篡改均fail-closed；

- 身份隔离：按设备ID、用户身份实现权限隔离，不同终端、用户未经授权无法访问彼此的上下文、内存、操作日志。

## 6.2 传输安全

- 全链路加密：终端与X-Hub之间的所有通信（上下文同步、AI请求）均采用TLS 1.3，仅两端可解密，中间节点无法窃听、篡改；

- 高风险动作签名清单：涉及支付、外发、不可逆写入等动作时，必须使用Hub生成并签名的`TxManifest/ActionManifest`（包含`intent_id`、目标对象、金额/参数、过期时间、策略标签）；终端只可展示与执行签名清单，不可信本地拼接数据；

- 一次性校验码（SAS）：终端展示高风险卡片时必须显示`Hub签名状态 + SAS`；确认端B需在本地验签后独立计算SAS并展示，用于跨终端人工交叉核验；

- 防重放与幂等：所有高风险请求必须携带`intent_id + execution_id + expires_at`；X-Hub对重复执行、过期清单、签名不匹配请求一律拒绝；

- 远程加密增强：远程公网模式建议使用WireGuard/ZeroTier/Cloudflare Tunnel等专用加密通道，并结合证书轮换与访问源限制；

- 局域网防护：通过虚拟局域网（VLAN）/防火墙/允许访问列表（allowed_cidrs）缩小X-Hub的可达范围，限制内网设备访问X-Hub的权限，弥补局域网内IP无法完全隐藏的短板。

## 6.3 存储安全

- X-Hub存储加密（🟡分阶段落地）：已完成Hub `turns.content` / `canonical_memory.value` 的AES-256-GCM at-rest envelope加密与KEK/DEK轮换，密文篡改默认fail-closed；待补齐Raw Vault/Observations/Longterm与终端本地`raw_log/skills/vault`（含Keychain root key托管与统一轮换作业）；

- 存储权限最小化：仅X-Hub的AI能力模块可读取上下文、内存，其他模块（如审计）需申请临时授权，且仅可读取脱敏后的数据；

- 原始证据策略：连接器写入Raw Vault时默认保留完整输入/输出证据并强制at-rest加密；可选配置短TTL暂存大体量正文，仍长期保留metadata与哈希证据链（建议“全文缓存”默认TTL为24小时）；

- Secrets策略：密钥、口令、Token统一加密托管于X-Hub；支持“禁远程外发Secrets”开关（开启后仅允许本地模型/本地工具消费）；

- 终端存储限制：普通终端自行保存自身对话的上下文（供AI读取作为历史内存），无其他永久存储；X-Terminal仅缓存最新几轮上下文，无永久存储，缓存可随时清理；

- 内存安全：X章程作为长期内存层的固定常驻条目，单独加密存储，仅可通过冷存储Token更新，确保不可篡改、永不下沉。

## 6.4 操作安全

- 全流程审计：X-Hub记录终端接入、上下文同步、AI请求、权限变更、技能编辑、内存升降级、泳道状态变化等关键事件，日志可检索、可追溯、可长期留存；

- 异常行为检测：实时识别高频敏感输入、异常上下文洪峰、证书/Token异常、异常并发漂移等行为并触发告警；

- 自动化受控执行：默认自动执行，但仅在授权与策略满足时放行；命中高风险规则时转入排队确认或阻断；

- 依赖双绿门控：跨泳道解阻必须同时满足`contract_green + runtime_green`，任一不绿即维持blocked（fail-closed）；

- 阻塞SLA升级：blocked任务按P0/P1/P2执行SLA计时与升级，超时必须自动升级到总控并留下机读证据；

- require-real证据纪律：发布与高风险链路禁止使用synthetic/smoke证据冒绿，样本不足必须返回`INSUFFICIENT_EVIDENCE`；

- 状态改写边界：禁止跨泳道直接改写他人任务状态，只能通过标准事件（unblock/guidance/handoff）驱动推进；

- 价值约束防护：X-Terminal在项目执行与AI决策时强制读取X章程，防止人类诱导越权与自动化价值漂移。

## 6.5 Hub签名Manifest（Hub-Signed Manifest）与跨终端校验码（SAS）

为实现“Hub是唯一可信源”，X-Hub将所有具有外部副作用的动作统一抽象为由Hub生成的Manifest对象，并由Hub签名。终端A/B/C均不得信任本地数据（剪贴板、缓存、UI渲染、系统时钟等），只能信任“Hub签名的Manifest”。

### 6.5.1 Manifest：以ETH转账TxManifest为例

- Hub生成TxManifest并签名；终端只接收、展示、执行，不得在终端侧重写金额、地址、Nonce或费用参数；

- `manifest_hash = SHA-256(CanonicalJSON(manifest))`；

- `hub_sig = Sign_Hub(manifest_hash)`；

- 关键字段（示例）：`intent_id, created_at, expires_at, action_type=eth_transfer, chain_id, asset, to, value, data, nonce, fee_params, gas_limit, policy_tags, required_grants, hub_pubkey_id`。

### 6.5.2 终端展示要求：Hub签名 + 一次性校验码（SAS）

- 终端展示必须包含：Hub公钥指纹/ID、签名验证结果、SAS一次性校验码；并支持展示/导出`manifest_hash + hub_sig`（复制或二维码）；

- SAS计算（建议）：`SAS = Base32(Truncate60(manifest_hash))`，按`XXXX-XXXX-XXXX`分组并带校验位，便于人工核对；

- 确认端B“独立校验码”：B在本地完成`hub_sig`验签后，使用同一算法独立计算SAS并展示；禁止直接展示Hub下发的SAS字符串（防止终端被攻破时欺骗用户）。

### 6.5.3 执行链路（A→Hub→B→Hub→C）

1. A发起“意图请求”（如“向Alice支付0.5 ETH”）；A提供的地址/金额视为不可信输入，仅用于Hub生成Manifest的参考；

2. Hub生成并签名TxManifest，同时推送给A与B；A/B展示地址、金额、SAS与Hub签名状态；

3. 用户在B确认（或扫描A的二维码比对SAS）后，Hub签发Grant（含TTL、额度、范围）；Grant可作为Pre-Grant提前获取（默认TTL=2h）；

4. Hub向C（X-Wallet）下发“执行命令 = TxManifest + Grant”；C必须再次验签，并构造与Manifest字段严格一致的交易，再由C的钱包密钥签名广播；

5. C回传`tx_hash`与回执；Hub落审计并将结果同步到A/B。

### 6.5.4 单签热钱包的剩余风险与缓解

当C为单签热钱包时，C一旦被攻破仍可能绕过策略直接签名恶意交易。X-Hub的缓解措施包括：专用设备/最小化依赖、额度与频率限制、白名单地址、强制B确认（对新收款地址/大额）、一键Kill-Switch、以及在路线图中支持硬件钱包/多签/阈值签名。

## 6.6 应急安全

- 全局紧急停机（Global Kill-Switch）：支持一键关闭所有终端接入、所有AI能力，秒级生效，应对重大安全风险；

- 权限回收：一键回收单个/多个终端的接入权限、AI调用权限，立即禁止终端与X-Hub通信；

- 数据清理：一键清理X-Hub中存储的所有上下文、内存数据，或单个终端的相关数据，避免数据泄露；

- 告警响应：异常行为触发告警后，管理员可快速查看告警详情、定位异常终端，采取应急措施（如停机终端、回收权限），实现实时止损。

# 八、应用场景

X-Hub系统凭借多模式适配、高安全等级、灵活的终端设计，可广泛应用于家庭、中小企业、高安全需求场景、机密科研、金融机构、工业控制等各类场景，聚焦AI交互的技术安全保障，具体如下：

## 8.1 家庭场景

适配模式0/1/2，满足家庭AI交互需求，平衡安全与体验：

- AI交互：儿童使用普通终端调用X-Hub AI能力进行学习咨询，终端无公网访问、无缓存，安全可控；家长使用X-Terminal进行多轮AI对话（如工作咨询、生活助手），还可通过X-Terminal的首页查看所有项目汇总进度，甚至运动时聆听进度播报并获取下一步指导；X-Terminal处理相关项目时，会自动遵循X章程约束，避免AI行为失控；

- 安全保障：终端无公网访问（模式1），避免儿童终端遭受网络攻击；X-Hub可通过隧道/中继等方式减少公网暴露，保障家庭内网安全可控；上下文沙箱隔离，防范恶意指令攻击X-Hub核心。

## 8.2 中小企业场景

适配模式1/2/3，满足企业AI办公需求，平衡效率与安全：

- AI办公：员工使用普通终端调用X-Hub AI能力（如文档汇总、会议纪要生成）；管理人员使用X-Terminal实现连续AI交互，可同时处理多个工作项目，并通过X-Terminal查看所有项目汇总进度；X-Terminal处理企业相关项目时，受X章程约束，防范人类诱导AI泄露企业敏感信息；

- 远程办公：出差人员通过模式3（远程公网加密连接）调用X-Hub AI能力，不影响办公效率且保障数据安全；

- 安全保障：通过虚拟局域网（VLAN）与防火墙配置缩小X-Hub的可达范围，降低内外网攻击风险，依托X-Hub全流程审计、异常行为检测，保障企业AI办公数据安全。

## 8.3 X-Terminal多模型Supervisor：一个AI管理多个进行中的项目（✅基础链路可用，🟡自治增强推进中）

典型场景：用户希望“只跟一个AI说话”，但项目会持续增减、优先级会动态变化，且希望系统在少打断前提下自动推进到高质量交付。

- 单入口多项目：用户只与Supervisor对话，Supervisor在后台托管多项目并行执行与状态汇总；

- 自动拆分与分配：Supervisor根据复杂度与模块图自动拆分`pool -> lane`，并按风险/预算/截止压力分配AI角色；

- 本地+付费混合编排：同一Run可并行调用本地模型与付费模型，付费能力受grant/quota约束并统一审计归因；

- 可调执行风格：支持`conservative/balanced/aggressive`三档拆分策略，支持`zero_touch/critical_touch/guided_touch`三档用户介入等级；

- 低Token高质量推进：每条泳道使用三段式提示词（Stable Core + Task Delta + Context Refs），默认只下发增量与引用；

- 反阻塞自动续推：Supervisor持续维护wait-for依赖图；当blocker满足dual-green后，等待泳道收到即时guidance并继续推进；

- 价值约束常驻：Supervisor在任务拆分、建议与执行阶段受X-Constitution约束，确保效率提升不突破安全与价值边界；

- 分层收口与发布：先池内集成测试，再全局mergeback与回滚校验，最后生成可审计交付摘要通知用户。

可验证进展（2026-02-25）：并发调度压测显示g2/p1/n8相较基线g1/p1/n8，queue_p90约下降65%，wall_p90约下降51%，验证“并行提速 + 门禁可控”可同时成立。

## 8.4 ETH加密资产支付：Hub签名TxManifest + SAS + 多端校验（A/B/C）（✅协议可用，🟡执行端安全持续增强）

落地链路：ETH优先。该场景的核心目标是：Hub是唯一可信源，终端不可信也不影响金额/地址/交易参数的安全。

- 终端角色：
  - 终端A（发起端）：发起支付意图与收款地址输入（不作为可信数据源）；
  - 终端B（确认端）：独立验签并显示独立校验码（SAS），作为人类最终确认入口；
  - 终端C（执行端）：联网的X-Wallet（可为单签热钱包），仅执行Hub签名的交易意图。

- 核心机制：Hub生成并签名TxManifest；终端展示必须包含Hub签名验证结果与SAS；确认端B必须本地独立计算SAS（详见6.5）。

- 执行流程：
  1) A向Hub提交“支付意图”（如：向Alice支付0.5 ETH）；
  2) Hub生成TxManifest并签名，推送给A与B；A/B展示金额、地址、SAS与Hub签名状态（支持二维码便捷核对）；
  3) 用户在B确认后，Hub签发Grant并下发给C；若已存在有效Pre-Grant（默认TTL=2h，且满足白名单/额度限制），可跳过执行时确认；
  4) C验签并构造与TxManifest字段严格一致的交易，签名并广播；回传`tx_hash`与回执给Hub；
  5) Hub全流程审计并同步结果至A/B。

- 风险与缓解：当C为单签热钱包时，仍需通过额度/频率限制、白名单地址、对新地址/大额强制B确认、Kill-Switch等降低剩余风险；路线图支持硬件钱包/多签/阈值签名进一步强化。


## 8.5 X-Terminal机器人线下买水：证据驱动的现实支付闭环（🟡场景化落地）

典型场景：用户让X-Terminal机器人“去买一瓶水”。系统目标不是“机器人会说话”，而是“现实动作可验证、可授权、可追责”。

- 步骤1（任务下发）：用户在终端A下发任务（商品偏好、预算上限、支付限制）；
- 步骤2（现场取证）：机器人到店后拍摄商品照片，读取价格并采集商户付款码，连同时间戳/位置摘要回传X-Hub；
- 步骤3（Hub生成动作意图）：X-Hub将“证据包 + 支付参数”封装为签名ActionManifest，并下发到用户手机终端B；
- 步骤4（跨端确认）：用户在手机端独立验签并核对SAS后确认支付；
- 步骤5（执行与回执）：Hub向执行端下发可执行指令，完成付款并回传结果，审计链路记录“谁发起、谁确认、何时执行、依据何证据”。

与纯加密资产支付场景的关键差异：该流程不仅校验金额和地址，还校验现实证据（照片/价格/付款码），把“现实世界不确定性”纳入可验证流程。

## 8.6 手机+蓝牙耳机Supervisor：无屏、低打断的人机协作（✅可落地）

典型场景：用户在通勤、运动、出差时不看屏幕，仅通过耳机持续掌握多个项目。

- 用户在手机连接蓝牙耳机后，与Supervisor进行语音对话；
- Supervisor按项目heartbeat播报“当前状态、风险变化、等待授权事项、建议下一步”；
- 用户用自然语音直接下达指令（继续/暂停/改优先级/授权某能力）；
- X-Hub将口头指令转为受策略约束的结构化动作，必要时触发Manifest确认；
- 多个Project可并行推进，用户始终只需面对一个“总调度入口”。

## 8.7 一眼看懂价值：从“会聊天”到“可托付执行”

- 对个人用户：你得到的不只是问答，而是“随时可接管、可审计、可确认”的自动化助理；
- 对企业团队：你得到的是“可并行、可治理、可追责”的AI执行基础设施，而不是难以审计的黑盒自动化；
- 对安全敏感场景：你得到的是“Hub可信核心 + 终端不可信 + Manifest/SAS + fail-closed”组合防线；
- 对长期项目：你得到的是“五层记忆 + X-Constitution”的稳定协作底座，减少上下文漂移与价值偏航；
- 对成本与效率：你得到的是“多AI分工 + 三段式提示词 + 反阻塞续推”的效率栈，在质量门禁下提升吞吐并降低Token浪费；
- 对开发者生态：你得到的是“源码可见 + Agent兼容 + 机读门禁证据”的可扩展协作平台。

## 8.8 开源协作治理：把“可运行”升级为“可共同演进”（✅首版就绪）

为保证系统不仅“能跑”，还能“被社区长期共建”，X-Hub将工程治理本身产品化：

- 发布透明：提供`CHANGELOG.md`与`RELEASE.md`，明确每次版本的能力增量、风险边界与回滚建议；
- 协作规范化：提供Issue/PR模板、CODEOWNERS与Dependabot，降低外部贡献的沟通成本与回归风险；
- 门禁可机判：将Gate/KPI/证据引用纳入工单与发布清单，避免“口头完成”与“不可复核”；
- 安全优先：高风险链路坚持fail-closed与require-real证据纪律，不以演示性通过替代发布质量；
- 生态兼容：以Agent Skill ABI兼容桥为例，兼容开源生态同时守住授权、审计与可回滚边界。
