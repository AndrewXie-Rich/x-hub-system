# 给 Agent 用户：X-Hub 的安全升级（不牺牲速度与控制力）

## 1）核心结论

X-Hub的目标不是削弱Agent能力，而是在保留“AI可读/可写/可执行”的体验前提下，把高风险能力收敛到可审计、可撤销、可冻结的控制面。

你仍然可以保持高自由度与高效率；差别在于关键风险点有了统一治理边界。

## 1.1）为什么这已经不只是概念

这套东西已经开始从“白皮书设想”进入“可验证现实”：

- 付费 GPT 类模型与本地模型已经开始进入同一 Hub 控制面；
- X-Terminal 已能真实显示配置模型、实际模型与降级结果，而不是表面成功；
- Supervisor 与项目执行链路也已进入可运行的早期阶段。

所以它的价值不再只是理论上更安全，而是已经可以被真实试用与验证。

## 2）信任模型

- **唯一可信源：** X-Hub。
- **默认不可信：** 所有终端（含本地缓存、剪贴板、UI展示结果）。
- **执行原则：** 高风险动作以Hub签名对象为准；终端本地拼接数据不作为可信依据。

## 3）普通终端接入 X-Hub 后，实际获得什么

在普通 Agent 终端模式下，默认仍保持高自由：

- 模型路由统一到Hub（本地模型 + 付费模型）；
- 付费API Key只在Hub加密存放，不落地终端；
- Hub可控制模型可用范围、Token预算、时长预算、是否允许联网/远程付费调用；
- 设备异常时，可立即切断该设备的AI能力与AI联网能力。

边界也要明确：

- 普通终端本地执行的系统动作（本地文件、脚本、邮件客户端本地发送等），Hub无法完全接管；
- 因为普通终端默认不使用Hub记忆，终端本地上下文被篡改/泄露的风险仍存在。

## 4）为什么 X-Terminal 安全性更高

X-Terminal在不牺牲体验前提下，新增的是“持续治理能力”：

- 记忆、项目状态、技能由Hub统一管理与审计；
- 本地仅保留最小缓存（非权威数据）；
- 支持多项目并行 + Supervisor统一汇报；
- 可以提前收集授权（Pre-Grant）避免执行时打断。

结果是：终端即使被攻破，影响主要局限在终端本地，Hub核心与跨项目资产仍可控。

## 5）Agent 的两种接入方式（自由优先，可选增强）

- 方式A：仅使用Hub模型能力（最大自由）
  - Agent终端继续本地执行业务动作；Hub主要负责模型、付费、配额与联网能力控制；
  - 适合追求零迁移、零流程变化的用户。

- 方式B：把高风险动作接入Hub控制面（可选）
  - 关键动作通过Hub统一授权、审计、撤销与紧急冻结；
  - 适合需要更强可审计性与团队治理的用户。

你可以先A后B，逐步迁移，不强制一次切换。

## 6）付费/Secrets/远程调用治理

- 付费授权粒度：`(device_id, user_id, app_id)`；
- 默认策略：首次人工确认一次，后续自动续签（可随时撤销）；
- 预授权（Pre-Grant）默认TTL：2小时；
- Secrets支持“禁止远程外发”模式（仅本地模型/本地工具可消费）。

## 7）给 Agent 用户的关键价值

- **不牺牲体验：** 保持Agent效率与自由度；
- **可控成本：** 付费模型与Token预算集中治理；
- **更强止损：** 设备异常可快速冻结相关能力；
- **可进化路径：** 从“模型托管”平滑升级到“动作治理与全链路审计”。

## 8）建议落地顺序

1. 先接入Hub模型路由与付费治理（最快见效）；
2. 再启用高风险动作的Hub授权与审计；
3. 长期项目迁移到X-Terminal，使用Hub记忆与Supervisor统一管理。

## 9）公开版本状态

X-Hub 当前仍是早期公开测试版，而不是完整商用品。

但这恰恰说明它现在值得关注：核心架构已经足够真实，产品形态又还没有完全固化，外部贡献者仍然能真正影响它最终会变成什么。
